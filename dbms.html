Understanding Data, Information, and Databases

Data refers to unorganized, raw facts or symbols gathered from various sources. On their own, these values or observations do not convey any specific meaning. Example: Numbers like 85, 90, 78—possibly student scores—are meaningless without further interpretation.

What is Information?

Information is data that has been processed and structured to provide relevance and meaning. It gives context to raw data, making it useful for decision-making. Example: "The class average score is 84.3" is meaningful information derived from the raw scores.

Characteristics of Good Information

Accuracy: Should be error-free and dependable. Relevance: Must relate directly to the subject or decision at hand. Timeliness: Should be current and available when needed. Completeness: Must include all necessary data. Clarity: Easily interpreted and clearly presented.

What is a Database?

A database is an organized collection of related data, stored electronically, allowing for efficient data retrieval, storage, and management. Example: A school's database may include student information, exam results, and attendance, all stored in a structured manner.

Purpose of a Database

Databases aim to systematically organize and manage large volumes of data for: Easy access and retrieval Secure and consistent data handling Minimizing redundancy Supporting multiple users and applications

Basic Database Terminology

Table: A grid of data organized in rows and columns. 1.Field: A column in a table representing a specific data attribute. 2.Record (or Tuple): A row in the table representing a single data item or entity. 3.Object: In object-oriented databases, it is a unit that includes both data and behavior. 4.Keys: i.Primary Key: Uniquely identifies each row in a table. ii.Foreign Key: Links one table to another using the primary key. iii.Candidate Key: Possible choices for a primary key. iv. Composite Key: A key made from multiple fields.

Data Dictionary

A data dictionary acts as a catalog that stores metadata about the database, including definitions for tables, fields, types, and constraints, helping manage the database design.

Types of Database Models

Hierarchical Model

  • Data is structured like a tree with one-to-many relationships.
  • Each child has one parent, but one parent can have many children.
  • Best for data with predictable hierarchy.

Network Model

  • Similar to hierarchical but allows many-to-many relationships.
  • Uses nodes and links to connect records.
  • More flexible but harder to design.

Relational Model

  • 1.Data is stored in tables.
  • 2.Relationships are established via keys.
  • 3.Supports SQL for queries.

Domain Constraint

  • Ensures values fall within a valid set for a field (e.g., age must be positive).

Entity Integrity

  • Requires each table to have a primary key that is unique and non-null.

Referential Integrity

  • Ensures foreign keys correspond to existing records in another table.

Key Summary

  • Primary Key: Uniquely identifies a row.
  • Foreign Key: Refers to a primar key in another table.
  • Candidate Key:Any column that can serve as a primary key.
  • Composite Key: Made up of more than one column.

Normalization

Normalization structures data to reduce repetition and improve integrity. It splits large tables into smaller ones linked by relationships.

Normal Forms

  • 1NF: Data is atomic; each field holds one value.
  • 2NF: Builds on 1NF; all non-key fields depend on the whole primary key.
  • 3NFBuilds on 2NF; eliminates transitive dependencies.

Benefits

  • Reduces redundancy
  • Enhances integrity
  • Simplifies updates
  • Improves storage efficiency
    • Avoids anomalies in insertion, update, or deletion

Limitations

  • Complex query writing due to multiple table joins
  • Slower performance for large-scale transactions
  • Risk of over-normalization
  • Design process can be time-consuming

Centralized Vs Distributed Databases

Introduction

  • Centralized DB: All data stored in a single location.
  • Distributed DB: Data is spread across several locationsa and coordinated as one system.

Benefits

  • Centralized: Easier to manage and back up.
  • Distributed Better fault tolerance, faster access for local users.

Database Security

Overview

Database security ensures that data is accessed only by authorized users and protected from threats like unauthorized access, data leaks, or cyberattacks.

Challenges

  • Preventing unauthorized access
  • Ensuring confidentiality and privacy
  • Defending against cyber threats
  • Managing user permissions

Security Techniques

  • Authentication: Confirms user identity.
  • Authorization: Grants specific access rights
  • Encryption: Secures data in storage and transit.
  • Audit logs: Track user actions.
  • Firewalls: Protects against unauthorized access.

DBA Responsibilities

  • Manages access rights and user roles.
  • Monitors and audit database activity
  • Apply updates and patches
  • Enforce compliance policies

SQL Datatypes

CHAR

The CHAR data type is used to store fixed-length character strings. When you define a CHAR column, you specify its length, and the system always reserves that exact amount of space. If the stored string is shorter, the remaining space is padded with blanks. This makes CHAR efficient for storing data where the length is consistent, such as country codes or fixed-format identifiers.

VARCHAR

VARCHAR stores variable-length character strings, which means it only uses as much space as the actual string requires, plus a small amount of overhead. This is more space-efficient than CHAR when storing strings of varying lengths, such as names or addresses. The maximum length must be defined when creating the column.

BINARY

BINARY is similar to CHAR but stores fixed-length binary data instead of characters. This is useful for storing data like images or encrypted information where exact binary representation must be preserved.

VARBINARY

VARBINARY stores variable-length binary data. It works like VARCHAR but for binary files or data streams. This allows for efficient storage when the size of the binary data varies.

TINYBLOB

TINYBLOB is a Binary Large Object designed to store very small amounts of binary data, such as thumbnails or small icons. It supports up to 255 bytes.

TINYTEXT

TINYTEXT stores small amounts of text data, up to 255 characters. It’s useful for brief text fields like short comments or tags.

TEXT

TEXT is intended for larger amounts of textual data, such as descriptions, articles, or notes. It can store up to 65,535 characters and is useful when you expect more extended content than VARCHAR can efficiently handle.

LONGTEXT

LONGTEXT stores very large amounts of text, up to 4 gigabytes in size. It is suitable for extensive documents, such as logs, HTML content, or long-form articles.

ENUM

ENUM defines a string column that can only take one value from a predefined list of allowed values. This ensures data integrity by restricting input to specific options, such as 'Male' or 'Female', or status values like 'Pending', 'Approved', or 'Rejected'.

BIT

BIT stores bit-field values and is commonly used to represent boolean flags or binary indicators where each bit has a specific meaning.

TINYINT

TINYINT is a very small integer type that typically occupies 1 byte and can represent values from -128 to 127 in signed mode or 0 to 255 if unsigned. It’s ideal for small numeric ranges, such as age or small counters. BOOLEAN

INTEGER (INT)

INTEGER or INT is a common numeric type used to store whole numbers. It typically uses 4 bytes and can represent large ranges of positive and negative numbers, suitable for general counting or identification purposes.

FLOAT and DOUBLE

FLOAT and DOUBLE are approximate numeric types that store decimal numbers using floating-point representation. FLOAT uses less storage but less precision, while DOUBLE uses more storage and provides higher precision. They are useful when exact decimal precision is not critical, such as scientific calculations.

DECIMAL

DECIMAL (or NUMERIC) is a fixed-point numeric type that stores exact decimal numbers with a defined precision and scale. It is often used for financial data where precise decimal representation is required to avoid rounding errors.

DATE

The DATE data type stores calendar dates (year, month, and day) without time. It is useful for birthdays, appointment dates, or any date-only information.

DATETIME

DATETIME stores both date and time values together, including hours, minutes, and seconds. This is useful for timestamps, event logs, or any data requiring full temporal detail.
